# 模式：故事库同步 story-sync

## 目标

从历史岗位评估报告、面试准备材料、面试记录、失败复盘和作品摘要中提取可复用的 STAR+R 故事线索，并回流到 `interview-prep/story-bank.md`。它不是生成“漂亮故事”，而是把已有事实整理成可追踪、可复核、可继续补全的故事资产。

## 输入材料

优先读取：

- `cv.md`：经历事实源；
- `article-digest.md`：作品、文章、案例和可证明成果；
- `reports/*.md`：岗位评估报告中的匹配亮点、简历建议、面试准备；
- `interview-prep/*.md`：面试准备材料和追问沉淀；
- `data/interviews.md`：真实面试记录；
- `data/failure-reviews.md`：失败复盘中“需要补进故事库”的内容；
- `interview-prep/story-bank.md`：已有故事库，用于去重和更新。

## 推荐命令

```bash
# 只生成候选故事预览，不改故事库
npm run story-sync

# 确认后追加到 interview-prep/story-bank.md
npm run story-sync -- --apply
```

## 处理流程

1. 扫描输入材料，提取包含项目、结果、协作、挑战、失败、复盘、AI 落地等信号的段落；
2. 按指标、动作、业务结果、能力标签、来源文件给候选故事打分；
3. 与 `story-bank.md` 中已有标题和证据来源做去重；
4. 生成 `STORY-DRAFT-xxx` 草稿，保留“待用户确认”的标记；
5. 只有在事实来源明确时才允许写入故事库；缺少 S/T/A/R 任一部分时，必须写成待补充，而不是编造。

## 输出结构

```markdown
# Story Sync 候选故事

| ID | 标题 | 能力标签 | 证据来源 | 置信度 | 建议动作 |
|---|---|---|---|---:|---|

### STORY-DRAFT-001｜{故事标题}

- **适用岗位：** 
- **适用问题：** 
- **能力标签：** 
- **可信证据：** {file_path}
- **状态：** 待用户确认事实边界

#### S｜背景
{从来源中可确认的背景；不确定则写“待补充”}

#### T｜任务
{从来源中可确认的任务；不确定则写“待补充”}

#### A｜行动
{从来源中可确认的行动；不确定则写“待补充”}

#### R｜结果
{从来源中可确认的结果；没有数字不要编造}

#### R+｜反思与复用
{可从复盘/面试追问中提炼；没有则写“待补充”}
```

## 质量规则

- 所有故事必须指向至少一个本地证据来源；
- 不生成不存在的指标、团队规模、预算、职级、客户名称；
- `--apply` 追加的是“草稿故事”，用户确认后才能改为正式 `STORY-xxx`；
- 如果同一项目在多个报告中反复出现，应合并为一个故事，而不是重复追加；
- 失败复盘也可以进入故事库，但要强调“学习与修正”，不能包装成虚假成功。
