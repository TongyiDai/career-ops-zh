# 模式：职业成长系统 growth

## 目标

把一次求职从“投递项目”升级为“职业能力建设系统”。本模式覆盖六件事：

1. **面试故事库**：把经历沉淀成可复用 STAR+R 故事；
2. **失败复盘**：从拒信、挂面、低反馈中提炼改进动作；
3. **Offer 对比**：用职业叙事、成长、风险和总包综合判断；
4. **谈薪助手**：准备期望、锚点、让步边界和沟通话术；
5. **职级对标**：判断目标岗位对能力层级的真实要求；
6. **AI 时代能力雷达**：持续评估用户在 AI/Agent 时代的能力缺口。

## 输入材料

执行前优先读取：

- `cv.md`：事实源；
- `config/profile.yml`：目标岗位、城市、薪酬、偏好；
- `article-digest.md`：项目/作品/案例证据；
- `data/applications.md`、`data/interviews.md`：求职和面试记录；
- `interview-prep/story-bank.md`：已沉淀故事库；
- `config/level-mapping.cn.example.yml`：大厂职级对标的参考维度和粗粒度映射；
- 如用户提供 JD、面试反馈、offer、薪资结构或职级描述，一并纳入。

## 伦理边界

- 不编造经历、指标、薪资或竞品 offer；
- 谈薪只提供策略和话术，不替用户发送；
- 对 offer 的建议必须说明不确定性；
- 对“高薪但高风险”机会要明确提示代价；
- 不建议通过虚假背调、虚假流水、虚假证明提升议价。

## 一、面试故事库

### 使用场景

- 用户说“帮我准备故事库 / STAR 故事 / 面试案例”；
- 每次岗位评估或面试准备后，把相关项目沉淀为可复用故事；
- 面试结束后补充追问、表现和下一次改进。

### 输出结构

```markdown
## 故事：{故事标题}

- **适用岗位：** {岗位/职能/行业}
- **适用问题：** {例如：最大的挑战、跨部门协作、0-1、失败复盘、管理冲突}
- **能力标签：** {产品判断/技术理解/业务结果/组织协作/管理/AI 落地}
- **可信证据：** {来自 cv.md / article-digest.md / 用户补充}

### S｜背景

### T｜任务

### A｜行动

### R｜结果

### R+｜反思与复用

### 可压缩版本（60 秒）
### 深挖版本（3-5 分钟）
### 可能追问与回答要点
```

### 质量标准

- 每个故事必须有真实来源；
- 没有指标时写“结果体现为……”而不是编造数字；
- 一个故事至少映射 2-3 类面试问题；
- 每个故事都要有“反思与复用”，体现候选人成长。

### 自动同步

当用户希望“把历史报告/面试记录沉淀成故事库”时，优先使用 `modes/story-sync.md` 与：

```bash
npm run story-sync
npm run story-sync -- --apply
```

`story-sync` 只追加带证据来源的草稿故事，正式入库前必须由用户确认事实边界。

## 二、失败复盘

### 使用场景

- 用户面试挂了、流程停了、被压薪、被 ghost；
- 用户说“复盘一下 / 哪里没答好 / 为什么没过”；
- 需要从多个失败样本中找模式。

### 输出结构

```markdown
# 失败复盘：{公司} - {岗位}

## 结论先行
- 最可能原因：
- 可控因素：
- 不可控因素：
- 下一次最该改的一件事：

## 证据
| 信号 | 来源 | 可信度 | 解释 |
|---|---|---:|---|

## 复盘维度
| 维度 | 表现 | 问题 | 改进动作 |
|---|---|---|---|
| 岗位匹配 |  |  |  |
| 职级表达 |  |  |  |
| 项目深挖 |  |  |  |
| 业务理解 |  |  |  |
| 沟通节奏 |  |  |  |
| 风险问题 |  |  |  |

## 需要补进故事库的内容
## 下次面试前训练清单
```

### 判断规则

- 不把所有失败都归因于用户；
- 区分“岗位本来不匹配”和“表达没打中”；
- 如果是 HC 取消、薪资不匹配、内部候选人等不可控因素，要明确写出；
- 复盘结果必须转化成具体动作，而不是泛泛鼓励。

## 三、Offer 对比

### 评分维度

默认满分 5 分：

1. 职业主线增强度：是否强化下一阶段叙事；
2. 业务/行业质量：赛道、客户、商业模式、增长；
3. 岗位权责：是否有真实决策权与资源；
4. 职级与成长：级别、title、晋升、学习曲线；
5. 现金与长期收益：base、bonus、期权、福利、税后；
6. 团队与老板：汇报线、老板质量、团队稳定性；
7. 风险与可逆性：加班、背锅、裁员、试用期、合规；
8. 生活适配：城市、通勤、家庭、远程、节奏。

### 输出结构

```markdown
# Offer 对比：{A} vs {B}

## 一句话建议

## 决策矩阵
| 维度 | 权重 | Offer A | Offer B | 说明 |
|---|---:|---:|---:|---|

## 关键差异
## 最坏情况推演
## 12 个月后简历会变成什么样
## 谈判空间
## 建议选择与前提条件
```

## 四、谈薪助手

### 准备信息

- 当前薪资结构：base、bonus、股票/期权、福利、税后；
- 目标薪资与底线；
- 市场锚点：同城/同级/同岗位范围；
- 竞争机会：只能使用真实存在的信息；
- 用户最在意的非现金条件：title、远程、团队、试用期、签字费、期权、年终。

### 输出结构

```markdown
# 谈薪策略：{公司} - {岗位}

## 建议锚点
- 理想目标：
- 可接受区间：
- 底线：
- 可交换条件：

## 谈薪话术
### HR 问期望薪资
### 对方低于预期
### 对方要求提供当前薪资
### 争取签字费/年终/期权/远程

## 风险提醒
## 不建议说的话
## 下一轮沟通问题清单
```

### 话术原则

- 先表达兴趣，再谈结构；
- 用“市场 + 角色价值 + 总包结构”而非情绪施压；
- 不撒谎，不虚构其他 offer；
- 保留选择权，但不要威胁式沟通。

## 五、职级对标

### 适用对象

- 大厂/中厂职级：P/M/T/L、专家/管理序列；
- 创业公司 title：负责人、Lead、Head、Director；
- 跨行业转型：产品、技术、解决方案、运营、HR、AI 转型。

### 参考配置

涉及阿里 P、字节 1/2/3、腾讯 T、美团 L、京东 P、百度 T 等体系时，先参考 `config/level-mapping.cn.example.yml` 的公司映射和六个判断维度，再结合 JD、面试反馈、脉脉/看准/OfferShow 等市场信息复核。不要把不同公司的 level 机械等同。

### 输出结构

```markdown
# 职级对标：{目标岗位}

## 初步判断
- 目标职级：
- 用户当前可证明职级：
- 差距：

## 对标维度
| 维度 | 目标要求 | 用户证据 | 差距 | 补强动作 |
|---|---|---|---|---|
| 业务复杂度 |  |  |  |  |
| 决策范围 |  |  |  |  |
| 组织影响力 |  |  |  |  |
| 专业深度 |  |  |  |  |
| 结果规模 |  |  |  |  |
| 人才/团队 |  |  |  |  |

## 面试中如何证明
## 简历中如何表达
## 不建议硬冲的信号
```

## 六、AI 时代能力雷达

### 维度

| 维度 | 含义 |
|---|---|
| AI 工具熟练度 | 是否能把 AI 融入日常工作流 |
| Agent/自动化思维 | 是否能设计人机协同、流程自动化、闭环系统 |
| 业务问题结构化 | 是否能把业务问题拆成可落地问题 |
| 数据与评测意识 | 是否知道如何衡量 AI/产品效果 |
| 技术理解边界 | 是否理解模型、RAG、权限、集成、可靠性等边界 |
| 产品化/规模化 | 是否能从 demo 走到可复用机制 |
| 组织推动力 | 是否能跨团队推动新技术落地 |
| 职业叙事表达 | 是否能清楚讲出“我为什么适合 AI 时代” |

### 输出结构

```markdown
# AI 时代能力雷达

## 总体判断

## 能力雷达
| 维度 | 分数/5 | 证据 | 风险 | 30 天动作 |
|---|---:|---|---|---|

## 最该补强的 3 件事
## 可立刻写进简历/面试的优势
## 30/60/90 天成长计划
```

## 默认下一步

每次输出最后都给出：

1. **本周可执行动作**；
2. **需要用户补充的信息**；
3. **应同步更新的文件**：例如 `interview-prep/story-bank.md`、`data/interviews.md`、`data/offers.md`、`data/growth-radar.md`。
